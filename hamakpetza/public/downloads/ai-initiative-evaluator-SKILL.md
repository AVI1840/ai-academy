# מנוע הערכת מזמיני AI — מסגרת מערך הדיגיטל

מנוע זה מבצע הערכה מהירה ומקיפה של כל מזמין / רעיון / פרויקט AI לפי מסגרת AI Journey של מערך הדיגיטל הלאומי. התוצר: **מסמך הכרעה מובנה** שמוכן לשיתוף עם מקבלי החלטות.

---

## הוראות שלב-אחר-שלב

### שלב 0: קבל תיאור המזמין

אם המשתמש לא סיפק תיאור — שאל:
> "תאר לי את הרעיון / הפרויקט / ההצעה שרוצים לבחון. כמה משפטים מספיקים — מה הבעיה, מה הפתרון המוצע, ומי הגיש?"

### שלב 1: נתח את המזמין

לפני שמתחיל ב-8Q — נסה להבין מהתיאור:
- **הדומיין:** back office / שירות אזרחים / ניתוח נתונים / אחר
- **רמת הדחיפות:** אם יש לוח זמנים — ציין
- **מי מגיש:** CDO / מנהל עסקי / ספק חיצוני / IT

---

### שלב 2: הפעל מסגרת 8Q

עבור כל שאלה — **הצג את השאלה**, **נתח את תיאור המזמין**, **הסיק תשובה ראשונית**, **ציין מה חסר**.

**Q1 — מה הבעיה העסקית?**
- האם הבעיה מוגדרת בצורה ברורה?
- האם יש מדד כמותי (זמן / שגיאות / עלות)?
- ציין: ✅ מוגדר / ⚠️ חלקי / ❌ לא מוגדר

**Q2 — מי המשתמש הסופי?**
- עובד פנימי / מנהל / אזרח / מערכת אחרת
- האם נדרש UI? נגישות? תמיכה בעברית?
- ציין: ✅ מוגדר / ⚠️ חלקי / ❌ לא מוגדר

**Q3 — מקור הנתונים**
- האם מקור הנתונים ידוע ומוגדר?
- האם יש בעלים (Data Owner)?
- האם הנתונים זמינים ואיכותיים?
- ⚠️ אם לא מוגדר — **חסם קריטי**, ציין בולט

**Q4 — Real-Time או Batch?**
- נסה להסיק מהתיאור: האם נדרשת תגובה מיידית?
- המלץ על הגישה הנכונה עם נימוק עלות-תועלת

**Q5 — פריסה: On-Prem / ענן ממשלתי / SaaS**
- האם יש Landing Zone קיים?
- האם הנתונים יכולים לצאת מהמשרד?
- המלץ: Landing Zone מרכזי / עצמאי / SaaS

**Q6 — מסגרת זמן**
- הסק מהתיאור או שאל:
  - POC: כמה שבועות?
  - Pilot: כמה חודשים?
  - Production: כמה חודשים?
- סמן אם הציפיות לא ריאליות

**Q7 — אינטגרציות**
- אילו מערכות קיימות צריכות להתחבר?
- זהה חסמי אינטגרציה צפויים (legacy systems, API חסר)

**Q8 — ROI**
- הסק מהתיאור או אמוד:
  - חיסכון שנתי משוער (שעות × עלות שעה)
  - עלות פיתוח + תחזוקה
  - תקופת החזר
- הערך: ✅ כדאי / ⚠️ שולי / ❌ לא כדאי

---

### שלב 3: BUY vs BUILD

צור טבלת החלטה:

| קריטריון | ניקוד BUY | ניקוד BUILD | משקל |
|---------|-----------|-------------|------|
| ייחודיות הצורך | | | 30% |
| רגישות נתונים | | | 25% |
| מהירות נדרשת | | | 20% |
| יכולת IT פנימית | | | 15% |
| תקציב זמין | | | 10% |

**המלצה:** BUY / BUILD / היברידי (BUY+BUILD) — עם נימוק ברור.

---

### שלב 4: בדיקת בשלות

**בשלות נתונים (1-5):**
- 1: Excel אישי / 2: מערכות ללא אחידות / 3: מרכזי עם בעלים / 4: DWH+API / 5: Real-time pipelines
- הסק מהתיאור ותן ציון + פירוט מה חסר

**בשלות ענן (1-5):**
- 1: ללא ענן / 2: ניסויים / 3: Landing Zone קיים / 4: פרויקטים בייצור / 5: DevOps מלא
- הסק ותן ציון

**מוכנות ארגונית (1-5):**
- 1: אין תמיכה / 2: מנהל אחד / 3: CDO/מוביל AI / 4: תקציב+צוות / 5: COE פעיל

## כללים חשובים

- **תמיד בעברית** — המסמך כולו בעברית, מונחים טכניים מותר באנגלית
- **Q3 חסר = חסם קריטי** — תמיד הדגש בולט אם מקור הנתונים לא מוגדר
- **אל תמציא נתונים** — אם מידע חסר, ציין "נדרש בירור" ולא ניחוש
- **מסמך ההכרעה מוכן לשיתוף** — פרמט כך שניתן ל-copy-paste לוואטסאפ / מייל / מצגת
- **ציין זמן הערכה** — כמה דקות לקח לנתח (מדגיש ערך)

## דוגמת שימוש

**קלט:**
> "מישהו הציע לנו AI שיסווג אוטומטית את פניות הציבור שמגיעות לדרשמ. יש לנו 500 פניות ביום, כרגע 3 עובדות עושות את זה ידנית, לוקח בין 2-5 דקות לפנייה."

**פלט מצופה:** מסמך הכרעה מלא עם:
- Q1 ✅ (בעיה מוגדרת: 1500-2500 דקות/יום ידני)
- Q3 ⚠️ (נדרש בירור: מאיפה מגיעות הפניות? מייל? מערכת?)
- המלצה: 🟡 עצור — הגדר מקור נתונים לפני POC
- ROI: חיסכון ~25 שעות עבודה/יום ≈ ₪300,000+ שנה
