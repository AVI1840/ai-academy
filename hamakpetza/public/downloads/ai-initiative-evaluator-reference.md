# Reference — מסגרת AI Journey המלאה

## מקור המסגרת
מסמך "AI Journey" של מערך הדיגיטל הלאומי, ממשלת ישראל.
המסגרת פותחה לסייע לדרשמים לבחון ולאמץ AI בצורה מובנית, מהירה ובטוחה.

---

## 8Q Framework — פירוט מלא

### Q1: הגדרת הבעיה העסקית

**שאלות עומק:**
- מה קורה היום ללא AI? (תאר תהליך ידני)
- כמה זמן לוקח? (מספרים)
- מה שיעור השגיאות? (אם רלוונטי)
- מי מושפע? (עובדים / אזרחים / שניהם)

**קטגוריות שימוש נפוצות:**
- **Back Office:** סיווג מסמכים, מיצוי טפסים, ניתוב פניות, עיבוד חשבוניות
- **Data Analysis:** דוחות אוטומטיים, זיהוי חריגות, חיזוי ביקוש
- **שירות אזרחים:** צ'אטבוט, ניווט זכויות, מענה שאלות
- **Visual Computing:** בדיקת תמונות, זיהוי שלטים, ניתוח סרטונים

### Q3: מקור הנתונים — בדיקת בשלות מלאה

```
שאלות חובה:
1. מה המקור? (DB / API / קובץ / מייל / ידני)
2. מי הבעלים (Data Owner)?
3. האם הנתונים עדכניים? (מתי עודכן לאחרונה)
4. האם הנתונים מדויקים? (שיעור שגיאות ידוע?)
5. האם יש הרשאות גישה?
6. כמה רשומות? (לאימון וניתוח)
7. האם נתונים אישיים? (DPO נדרש)
```

**רמות בשלות נתונים:**
1. Excel אצל כל עובד — **חסם: לא ניתן להמשיך**
2. מערכות קיימות, ללא אחידות — **נדרש Data Engineer**
3. מרכזי, בעלים מוגדר, API זמין — **מינימום לPOC**
4. DWH / Data Lake — **אידיאלי**
5. Real-time pipelines — **מוכן לייצור**

### Q5: פריסה — מפת אפשרויות ממשלתית

```
SaaS ← → On-Prem
       ↑
   נימבוס (AWS/GCP)
   ├── Landing Zone מרכזי (מ"דט)
   │   ├── מהיר להקמה
   │   ├── אבטחה מוגדרת מראש
   │   └── תיאום עם מ"דט נדרש
   └── Landing Zone עצמאי
       ├── גמישות מקסימלית
       ├── אחריות אבטחה על הדרשמ
       └── נדרש צוות IT מוסמך
```

**כלל אצבע:**
- נתונים לא רגישים, תחילת דרך → Landing Zone מרכזי
- נתונים רגישים / מסווגים → Landing Zone עצמאי
- כלים כלליים (Gemini Workspace, Copilot) → SaaS מאושר

## BUY vs BUILD — מטריצה מורחבת

| קריטריון | BUY | BUILD | היברידי |
|---------|-----|-------|---------|
| ייחודיות | גנרי | ייחודי | API+UI |
| נתונים | לא רגישים | רגישים | מעורב |
| מהירות | דחוף | לטווח ארוך | POC מהיר |
| צוות | קטן | גדול | בינוני |
| תקציב | מוגבל | גמיש | מוגדר |
| שליטה | נמוכה | מלאה | חלקית |

**המלצות לפי קטגוריה:**
- **Chatbot כללי:** BUY (Azure Bot / Google Dialogflow) + BUILD integration
- **ניתוח מסמכים פנימיים:** BUY API (OpenAI/Claude) + BUILD wrapper
- **מערכת החלטות משפטיות:** BUILD (נדרשת שליטה מלאה)
- **תמלול ישיבות:** BUY (Gemini / Azure Speech) + BUILD integration

## ROI — נוסחאות לחישוב

```
חיסכון בזמן:
שעות_חסכון/יום × ימי_עבודה × עלות_שעה

דוגמה: 3 שעות/יום × 220 ימים × ₪100/שעה = ₪66,000/שנה

חיסכון בשגיאות:
(שיעור_שגיאות_לפני - שיעור_שגיאות_אחרי) × עלות_תיקון_שגיאה × נפח

עלות AI:
פיתוח + רישוי_שנתי + תשתית_שנתית + תחזוקה + הכשרה

ROI = (חיסכון_שנתי / עלות_שנתית) × 100
תקופת_החזר = עלות_פיתוח / חיסכון_שנתי (בחודשים)
```

**סטנדרטים ממשלתיים:**
- ROI > 150% — ✅ מומלץ
- ROI 100-150% — ⚠️ שולי, בחן גם ROI לא כספי
- ROI < 100% — ❌ לא כדאי (אלא אם ROI איכותי ברור)

## לוחות זמנים ריאליסטיים

```
POC (הוכחת היתכנות):
└── 4-8 שבועות
└── צוות: 2-3 אנשים
└── תקציב: ₪50,000-150,000
└── מטרה: האם זה עובד טכנית?

Pilot (תוכנית ניסיון):
└── 2-4 חודשים
└── 5-20 משתמשים אמיתיים
└── תקציב: ₪150,000-400,000
└── מטרה: האם זה עובד בפועל?

Production:
└── 6-12 חודשים מה-POC
└── פריסה מלאה
└── תקציב: תלוי היקף
└── מטרה: ROI אמיתי
```

## שאלות CISO — 8 שאלות אבטחה

לכל פרויקט AI בממשלה, CISO בוחן:

1. **מה ה-UI?** — ממשק ויזואלי / API / מותקן?
2. **אוכלוסיית משתמשים?** — כמה? מי? הרשאות?
3. **מה מקור הנתונים?** — רגישות, מיקום, הצפנה?
4. **אחסון תוצאות?** — היכן? כמה זמן?
5. **Real-Time או On-Prem?** — האם נתונים יוצאים?
6. **אינטגרציות?** — מה מתחבר? API? Legacy?
7. **Interoperability?** — פרוטוקולים, תאימות?
8. **ROI vs סיכון?** — האם הסיכון מוצדק?
